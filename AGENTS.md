# 项目协作与开发规范

本文件适用于整个仓库。所有参与本项目的智能代理、开发者和自动化工具均须遵守。

## 1. 用户沟通与需求理解

- 默认使用简体中文沟通，除非用户明确要求其他语言。
- 用户不需要掌握技术术语。用户可以使用口语、业务语言、不完整描述或不准确名称提出需求。
- 收到需求后，必须先以产品经理视角识别用户真正要完成的工作、使用场景、输入、输出、限制条件和验收结果，不能机械地按字面实现。
- 不得贬低、嘲讽或复述用户对自己的负面评价。应保持尊重、耐心和直接。
- 发现名称错误、逻辑冲突、缺失条件或明显风险时，应主动修正并用通俗语言说明。
- 能根据项目上下文作出安全合理判断时直接推进；只有会显著改变产品范围、数据安全或使用结果的选择才询问用户。
- 必须询问时一次只问一个关键问题，并优先给出推荐选项、推荐理由和其他选项的影响。
- 技术方案必须翻译成用户能判断的产品影响，例如体积、速度、是否联网、是否需要管理员权限、客户现场是否可用。
- 不要求用户重复已经提供过的信息。新要求默认叠加到现有要求，除非用户明确表示替换。

## 2. 产品目标

本项目开发一款运行于 Windows 的等级保护测评辅助工具，用于在用户已获授权的设备和系统上：

1. 管理客户、项目、设备和连接资料。
2. 通过 SSH、Telnet、串口、密钥、跳板机、PowerShell、WinRM 等方式建立连接。
3. 使用低风险只读方式自动识别对象类别、厂商、产品系列、型号、操作系统、数据库、中间件和版本，识别不可靠时再请求人工确认。
4. 根据识别结果匹配测评命令。
5. 自动执行命令，处理分页、提权、交互提示、超时和失败。
6. 同时保存原始输出、结构化执行记录和可阅读截图。
7. 按项目、设备和测评项自动整理证据目录。
8. 通过可扩展命令包和连接插件逐步支持网络设备、服务器、数据库和中间件。

工具是“测评辅助工具”，不能代替测评人员作出最终合规结论。

## 3. 已确认的产品约束

- 正式目标平台为 Windows 10 和 Windows 11；后续 Windows 版本发布后再进行正式兼容性验证。
- 不再要求兼容 Windows 7。
- 首选轻量技术路线：C#、WPF、.NET Framework 4.8、Fluent 风格界面、PuTTY/Plink、SQLite。
- 界面必须现代、清晰、适合非技术用户，不得因为轻量化退回传统工具式粗糙界面。
- 主程序与大型可选组件分离。核心绿色版目标体积为 20–40 MB，标准安装包目标体积为 30–60 MB；最终以可重复构建后的实测体积为准。
- 提供轻量绿色版和现场完整包。完整包可附带常用离线依赖，但主程序保持一致。
- 缺少环境或组件时，必须说明缺少项、影响功能、安装大小、管理员权限、联网要求和解决办法。
- 自动安装任何环境或组件前必须获得用户确认；不得擅自静默安装、降低安全配置或重启电脑。

## 4. 技术与架构原则

- 使用清晰模块边界，至少分为：桌面界面、项目数据、凭据保护、连接适配器、会话执行器、命令包、证据生成、依赖检测和诊断日志。
- 界面与连接协议解耦，命令定义与执行引擎解耦，设备识别与命令匹配解耦。
- SSH、Telnet 和相关密钥工具优先复用 PuTTY/Plink 的成熟能力，不直接在 PuTTY 旧界面上堆叠业务功能。
- 特殊连接方式、数据库驱动和中间件适配器采用按需插件，不得全部塞入主程序。
- 本地业务数据使用 SQLite。证据文件保存在用户选择的项目目录，数据库只保存索引、元数据和校验值。
- 优先离线可用。联网功能必须是增强能力，不能成为基本测评流程的强制依赖。
- 不引入仅为开发方便但显著增加安装体积、启动时间或现场依赖的框架。
- 新增第三方依赖前必须检查许可证、维护状态、Windows 兼容性、安全风险和发布体积。

## 5. 命令库规则

- “只读查询”是不可放宽的产品安全约束，不是默认选项。软件自动发出的所有命令都只能读取信息，不得修改设备、系统、数据库、中间件、容器、文件、账户、网络或运行状态。
- 任何命令都必须标明：对象类别、厂商、产品系列、型号范围、版本范围、测评项、执行账户要求、风险等级、超时、分页处理和结果说明。
- 命令匹配优先级为：精确版本与型号、产品系列与版本、厂商通用、类别通用。不得把厂商通用命令冒充为所有型号均适用。
- 用户提供的命令先进入待校验区。允许纠错和补全，但不得凭记忆编造设备命令或数据库语句。
- 支持用户后期手动新增、复制、修改和导入命令，但新命令默认处于“待校验、禁止自动执行”状态；通过只读审查和版本验证后才能发布到正式命令包。
- 校验命令时优先使用厂商官方文档、官方知识库和可复现测试环境，并记录来源、验证日期和验证状态。
- 无法确认版本兼容性时必须标为“需要人工确认”，不能自动执行高风险命令。
- 自动任务只能从已签名或已校验的只读命令白名单中选择，不能把用户任意输入直接交给 shell、网络设备终端或数据库执行。
- 禁止命令包括但不限于：配置变更、文件写入或删除、服务启停或重启、进程终止、账户变更、权限变更、审计策略变更、数据增删改、结构变更、备份恢复、导入导出到服务器文件、容器启停或变更，以及可能触发这些副作用的脚本、函数和存储过程。
- 不能只靠危险关键词黑名单判断安全。必须同时使用正式命令白名单、对象和版本匹配、命令结构检查、数据库专用语法规则、受限参数模板、执行前预览、最小权限账户和审计记录。
- 对数据库而言，`SELECT` 字样本身不代表安全。可能写文件、调用有副作用函数、执行动态 SQL、加锁或改变会话外状态的查询也必须禁止。
- 连接后只允许先执行固定的低风险识别命令。对象类型或版本不确定时，不得尝试多个厂商命令包轮流执行，应停止并提示人工选择。
- 命令包必须支持独立版本、导入导出、签名或哈希校验、回滚和项目级锁定，避免测评过程中自动变化。

## 6. 自动识别规则

- 新建设备时默认提供“自动识别”类别，也允许用户提前选择网络设备、服务器、数据库、中间件或其他类别以缩小识别范围。
- 自动识别分为被动信息、固定低风险探测和人工确认三步。优先读取连接握手、服务横幅和已有系统信息，再执行最少量只读识别命令。
- 默认不得对客户网络进行大范围端口扫描、漏洞扫描或无关资产发现。探测范围仅限用户明确提供的主机、端口、登录会话和授权范围。
- 网络设备通过连接横幅、提示符和经过校验的版本查询识别厂商、设备类别、型号与系统版本。
- 服务器通过只读系统信息识别 Windows、Linux、发行版、内核和架构。
- 数据库发现分为主机发现和数据库直连。主机发现可只读检查已安装软件包、服务、进程、监听信息及 Docker/Podman 容器元数据；数据库直连再使用厂商安全元数据查询确认产品、版本和实例信息。
- 中间件可只读检查服务、进程、软件包、安装目录元数据、配置只读信息和容器镜像元数据，再用安全版本查询确认。
- 容器发现只能查看容器列表、镜像、标签、端口和只读元数据，不得启动、停止、创建、删除或修改容器。进入容器执行识别命令必须使用正式只读模板并记录。
- 每个识别结果必须保存识别依据和可信度。结果冲突、版本未知、发现多个实例或可信度不足时，必须列出候选项让用户选择。
- 未经用户确认的低可信识别结果不得触发完整命令包执行。

## 7. 安全与授权

- 仅对用户明确授权的客户、设备和系统执行连接与采集。
- 凭据不得明文落盘。使用 Windows DPAPI 保护密码、私钥口令和连接密钥。
- 日志、错误提示、截图元数据和导出诊断信息必须遮盖密码、令牌、私钥及敏感连接字符串。
- SSH 首次连接必须展示并确认主机指纹；指纹变化必须阻止自动连接并明确告警。
- 不得默认启用不安全算法。兼容老设备需要降低算法等级时，必须按设备单独授权并显示风险。
- 应建议并优先使用只读或最小权限测评账户。即使账户具备修改权限，软件仍必须阻止非只读命令。
- 自动采集界面不提供可绕过白名单的自由命令输入框。需要外部人工终端时必须与自动采集明确分离，并提示其操作不受本软件只读策略保护。
- 外部组件只能从可信官方来源获得，下载后校验数字签名或固定哈希。
- 不自动上传客户资料、命令输出、截图、设备信息或凭据。任何云端能力均须单独设计并获得明确同意。

## 8. 证据与可追溯性

- 每次执行至少记录项目、设备、连接方式、命令包版本、命令、开始与结束时间、退出状态、原始输出和错误信息。
- 截图必须来源于实际保存的终端输出，不得生成与原始输出不一致的内容。
- 长输出应分页生成连续截图，并带设备名称、测评项、时间和页码。
- 原始输出与截图计算 SHA-256，支持完整性检查。
- 目录和文件名需处理 Windows 非法字符、重名、路径过长和中文编码问题。
- 失败、跳过、人工确认和重新执行都必须可追溯，不能用成功截图掩盖失败记录。

## 9. 依赖检测与安装

- 使用轻量启动检查器处理主程序启动前的系统版本、.NET Framework 和文件完整性检查。
- 主程序内部按功能检测 PuTTY 工具、串口能力、PowerShell/WinRM、数据库客户端或驱动及其他插件。
- 非必需组件缺失时只禁用对应功能，不得阻止整个软件启动。
- 缺失提示必须提供“立即安装、选择离线安装包、查看人工步骤、暂不安装”四类处理能力；不适用的选项可以隐藏。
- 优先搜索软件旁的“依赖组件”目录和用户指定位置；联网下载只能使用官方地址。
- 安装过程应记录日志、检查退出码并在安装后重新检测。失败提示必须给出可执行的解决办法。
- 自动安装前显示来源、版本、体积、权限和影响，并由用户确认。

## 10. 界面与体验

- 核心导航围绕“客户/项目、设备、采集任务、证据、命令库、组件中心、设置”组织。
- 首次使用提供简短向导，不要求用户理解协议参数；高级参数折叠显示。
- 设备类别默认可选“自动识别”。识别页面必须展示发现依据、可信度、多个候选实例和人工修正入口。
- 连接页先提供常用字段，端口、编码、分页符、代理、算法等高级设置按需展开。
- 执行过程中持续显示当前设备、当前测评项、当前命令、耗时和状态，并允许安全停止。
- 错误信息使用“发生了什么、可能原因、怎么处理”的格式，技术详情放在可展开区域。
- 支持浅色和深色模式、高 DPI、键盘操作和清晰焦点状态。

## 11. 开发与验证

- 先实现可验证的最小闭环：建立项目、添加设备、连接、匹配命令、执行、保存原始输出和截图、查看证据。
- 新增功能优先编写可自动验证的核心逻辑测试；协议交互使用可控模拟服务和脱敏样本。
- 必须建立危险命令拒绝测试集，覆盖网络设备配置命令、Shell 写入与管道组合、PowerShell 状态变更、SQL 数据或结构修改、有副作用函数、服务和容器变更。
- 涉及真实设备的验证必须说明设备类型、版本、授权范围和测试结果，不得把单一型号测试推断为全品牌兼容。
- 发布前至少验证 Windows 10 与 Windows 11、中文路径、无管理员权限、无网络、错误凭据、网络中断、长输出和重复执行。
- 不宣称“支持所有品牌、所有版本”或“命令百分之百正确”。支持范围必须由已验证的适配清单证明。
- 修改应聚焦当前需求，不顺带重构无关模块；不得覆盖用户已有修改。
