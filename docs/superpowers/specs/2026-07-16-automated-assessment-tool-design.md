# 等级保护自动化测评工具产品设计

日期：2026-07-16
状态：待用户审阅

## 1. 产品定位

本产品是一款轻量、离线优先的 Windows 桌面工具，帮助等级保护测评人员在已授权的网络设备、服务器、数据库和中间件上执行只读采集命令，并自动整理原始输出和截图证据。

它解决的核心问题不是“替用户做出合规结论”，而是减少重复登录、查找命令、逐条输入、手工截图和整理目录的工作量，同时保留人工判断和复核入口。

## 2. 目标与非目标

### 2.1 首要目标

1. 用户可以用业务语言管理客户、项目和设备，无需理解复杂协议参数。
2. 软件默认自动识别设备或软件类别、厂商、型号、安装形态和版本，识别不可靠时再请求人工选择。
3. 软件能根据识别结果选择经过校验的命令。
4. 软件能批量执行只读命令并可靠处理分页、超时和交互提示。
5. 每次执行同时保存原始输出、执行记录和截图证据。
6. 软件可以通过 U 盘或普通文件传输工具方便地带到客户现场。
7. 客户现场缺少环境时，软件能检测、解释并协助安装。

### 2.2 首版不包含

- 不自动修改客户设备配置。
- 不对客户网络进行默认的全网端口扫描、漏洞扫描或未授权资产发现。
- 不自动给出最终测评结论或替代测评报告签字。
- 不在首版实现安全设备网页界面自动操作。
- 不在首版承诺覆盖所有品牌、型号、数据库和中间件版本。
- 不依赖云端服务，不自动上传客户数据。

## 3. 目标平台与发布形式

### 3.1 正式支持平台

- Windows 10
- Windows 11
- 后续 Windows 版本发布后，通过兼容性测试再加入正式支持清单

### 3.2 发布形式

提供两种分发包，但使用同一套主程序：

1. **轻量绿色版**：只包含主程序、基础命令包和必要的 PuTTY 工具，适合快速复制。
2. **现场完整包**：在绿色版之外附带常用离线依赖安装包，适合无网络环境。

核心绿色版目标为 20–40 MB，标准安装包目标为 30–60 MB。体积目标需在首个可运行原型完成后实测；若超出，必须列明各组件占用并优先拆分可选依赖，不能通过删除安全检查或证据功能来压缩。

## 4. 技术路线

### 4.1 基础技术

- 语言：C#
- 桌面框架：WPF
- 运行环境：.NET Framework 4.8
- 视觉：轻量 Fluent 风格主题和自有控件样式
- 本地数据：SQLite
- SSH/Telnet 工具链：PuTTY/Plink、PSCP、PSFTP、PuTTYgen、Pageant 中按需选用
- 凭据保护：Windows DPAPI
- 安装包：Inno Setup

### 4.2 选择理由

WPF 与 .NET Framework 4.8 在目标 Windows 系统上成熟稳定，不需要给主程序捆绑完整的现代 .NET 运行时。PuTTY 工具链承担复杂协议兼容，主程序专注项目、任务、命令和证据管理。可选数据库驱动与特殊适配器单独分发，以控制主程序体积。

### 4.3 不直接修改 PuTTY 界面

PuTTY 保留为底层连接工具和人工操作后备。业务界面由本产品独立实现，避免把项目管理、命令库、任务编排和证据管理耦合进 PuTTY 的传统界面代码。

## 5. 系统模块

### 5.1 桌面界面

负责项目导航、设备编辑、连接测试、任务执行、证据查看、命令库管理和组件管理。界面只调用应用服务，不直接执行协议命令或访问数据库文件。

### 5.2 项目数据服务

负责客户、项目、设备、任务、命令包版本和证据索引。SQLite 保存结构化数据，原始输出和截图保存在项目证据目录。

### 5.3 凭据保险箱

使用 DPAPI 加密密码、私钥口令和敏感连接参数。默认绑定当前 Windows 用户。界面仅在连接时按需解密，不向普通日志传递明文。

### 5.4 连接适配器

所有连接方式实现统一接口，首批包括：

- SSH 密码登录
- SSH 私钥登录
- Telnet
- 串口
- Windows PowerShell/WinRM
- SSH 跳板机或代理

后续数据库直连和特殊厂商协议通过插件加入，不修改任务执行器。

### 5.5 会话执行器

负责打开会话、识别提示符、执行命令、处理分页、等待输出、响应受控交互、判断超时和安全停止。它保存未经美化的原始字节或文本输出，并向界面发布进度事件。

### 5.6 设备识别与命令匹配

连接成功后先读取连接握手、服务横幅和已有信息，再执行最少量固定只读识别命令，提取对象类别、厂商、型号、系统名称、安装形态和版本。用户可修正识别结果。匹配顺序为精确型号和版本、产品系列、厂商通用、类别通用；每次匹配都显示依据和可信等级。低可信、冲突或未知结果不能触发完整命令执行。

### 5.7 命令包管理

命令包是独立版本化文件，包含命令定义、适用范围、风险、输出处理和来源信息。项目开始执行后锁定命令包版本，更新命令库不会悄悄改变已开始项目的执行内容。用户可以后期手动补充不同版本命令，但新增或修改内容先进入待校验区，通过只读安全审查和版本验证后才能发布。

### 5.8 证据生成器

把实际原始输出转换为统一终端样式截图。长输出生成连续分页图片；图片包含项目、设备、测评项、命令、执行时间和页码。原始输出和截图均计算 SHA-256。

### 5.9 组件中心

集中显示必需和可选组件的状态、版本、来源、用途和安装操作。组件安装完成后立即重新检测，不要求用户自行猜测是否生效。

### 5.10 诊断日志

记录启动检查、连接阶段、命令执行阶段、组件安装和证据写入错误。默认脱敏；用户可以导出诊断包，但凭据、完整私钥和客户命令输出默认不进入诊断包。

## 6. 自动识别设计

### 6.1 总体策略

新建设备时默认选择“自动识别”，用户也可以提前指定网络设备、服务器、数据库或中间件来缩小范围。识别按以下顺序进行：

1. 被动读取连接握手、横幅、提示符和用户已填写的信息。
2. 在当前授权会话内执行固定的低风险只读识别命令。
3. 汇总识别依据，计算高、中、低可信度。
4. 高可信且结果唯一时自动采用；中低可信、结果冲突或多实例时让用户确认。
5. 确认对象和版本后才加载对应完整命令包。

软件不会为了猜测类型而把多个厂商的命令依次发送给目标，也不会默认扫描用户未提供的其他主机或端口。

### 6.2 网络设备

通过 SSH/Telnet 横幅、提示符和经验证的版本查询识别厂商、交换机或路由器类别、型号及系统版本。若设备屏蔽版本信息或返回结果与命令包不匹配，显示候选厂商和型号供人工选择。

### 6.3 服务器

通过固定只读系统信息识别 Windows 或 Linux、发行版、系统版本、内核和架构。服务器识别完成后，可以继续发现其上安装的数据库和中间件，但发现范围仅限当前已授权主机。

### 6.4 数据库

数据库识别分为两种入口：

1. **主机发现**：用户先通过 SSH 或 WinRM 登录服务器，软件只读检查已安装软件包、Windows 服务、Linux 服务、进程、已知监听信息以及 Docker/Podman 容器元数据，形成可能的数据库实例列表。
2. **数据库直连**：用户提供数据库地址和凭据，软件通过协议握手和厂商允许的安全元数据查询确认数据库产品、版本、实例和必要运行信息。

本地安装、系统服务安装和容器安装均显示安装形态。容器发现只读取容器列表、镜像、标签、端口和元数据；不得启动、停止、创建、删除或修改容器。发现多个实例、无法确定厂商、版本缺失或直连失败时，提示用户选择或补充连接信息。

### 6.5 中间件

使用与数据库相同的“主机发现加服务确认”模式，只读检查服务、进程、软件包、安装元数据和容器镜像信息，再执行厂商验证过的版本查询。Nginx、IIS、Tomcat、Apache 和后续国产中间件分别使用独立适配包。

### 6.6 安全设备

命令行型安全设备可以使用网络设备识别流程；主要依赖网页管理的设备首版只允许用户手动登记类别、品牌和版本，不进行网页自动操作。

## 7. 核心用户流程

### 7.1 建立项目

用户新建客户和项目，选择证据根目录。软件检查目录可写、剩余空间、路径长度和是否存在同名项目。

### 7.2 添加设备

设备类别默认为“自动识别”，用户也可以提前选择类别。用户填写便于识别的名称、地址和登录资料。常用参数直接显示，高级参数按需展开。支持从模板或表格批量导入，但导入后必须校验必填项和重复设备。

### 7.3 测试连接

软件依次检查地址、端口、协议工具、主机指纹和身份验证。失败时按“发生了什么、可能原因、怎么处理”显示，不只展示技术错误码。

### 7.4 识别与确认设备

软件先被动识别，再执行固定只读识别命令，并展示类别、厂商、型号或软件产品、安装形态、版本、识别依据和可信度。高可信且唯一的结果可自动接受；结果不完整、冲突或存在多个实例时要求用户从候选项中确认，不能猜测后继续批量执行。

### 7.5 生成采集任务

软件根据设备信息和选定测评范围生成命令清单。用户可查看每条命令的用途、风险和适用版本，可取消不需要的命令。首版只允许自动执行只读命令。

### 7.6 执行与监控

任务界面显示设备、测评项、当前命令、已用时间、成功、失败、跳过和待人工确认数量。用户可安全停止；停止后保留已完成证据，并清楚标记未执行项。

### 7.7 查看和导出证据

证据页按设备和测评项展示原始输出、截图、执行状态和完整性校验。用户可以重新执行单项、补充人工截图、添加备注，并导出项目证据目录。

## 8. 证据目录设计

默认目录结构为：

```text
客户名称/
  项目名称/
    设备名称_IP或地址/
      测评项编号_测评项名称/
        执行记录.json
        原始输出.txt
        证据_001.png
        证据_002.png
```

文件名中的 Windows 非法字符自动替换。名称过长时保留可读前缀并追加短哈希，避免因路径过长写入失败。重复执行创建新的执行批次，不覆盖旧证据。

## 9. 命令包设计

每条命令至少包含以下信息：

- 唯一标识和测评项
- 对象类别、厂商、产品系列和型号范围
- 最低与最高适用版本或明确版本集合
- 命令文本和字符编码
- 是否需要特权模式及进入、退出方式
- 超时时间、结束提示符和分页处理
- 是否只读及风险等级
- 输出中应遮盖的敏感字段
- 官方来源或验证记录
- 验证状态和最后验证日期

同一测评项可以包含多条互补命令，不会因为测评项名称相同而被合并。只有明确设置同一 `alternativeGroup` 的命令才表示“完成同一用途的不同厂商、型号或版本实现”；匹配时仅在该组内选择适用范围最精确的一条，精确度相同时按命令包中的先后顺序选择，避免把通用命令和精确命令重复发送到客户设备。

用户提供的命令导入待校验区。系统可以检查重复、明显语法问题和版本冲突，但“自动补全”只生成建议，必须通过官方资料或真实测试确认后才能进入自动执行命令包。

用户可以在图形界面手动新增不同品牌、型号和版本的命令，也可以从文件批量导入。所有新增或修改命令均保存编辑历史，默认禁止执行；只有完成只读安全审查、适用版本校验和测试记录后，才能发布为正式命令包。

## 10. 首批支持范围

首个可用版本聚焦闭环质量，不追求一次覆盖全部对象：

1. 网络设备：选择两到三个主流厂商的交换机和路由器系列作为验证样本。
2. Linux 服务器：通用 shell 采集，覆盖常见发行版差异识别。
3. Windows 服务器：PowerShell/WinRM 和可用时的 SSH。
4. 命令包导入：支持用户现有命令整理和待校验工作流。

数据库和中间件从第二阶段开始按适配包逐个加入。每个适配包必须明确驱动要求、版本范围和验证设备，不能用“主流数据库全部支持”作为验收描述。

## 11. 依赖检测与自动安装

### 11.1 启动前检查

一个独立轻量启动器在主程序运行前检查：

- Windows 版本是否满足最低要求
- .NET Framework 4.8 是否可用
- 主程序关键文件是否完整
- 当前目录和项目目录是否可写

如果主程序无法启动，启动器仍能显示中文修复界面。

### 11.2 功能级检查

主程序按用户实际使用的功能检查 Plink、串口、PowerShell/WinRM、数据库客户端和插件。缺少可选组件只禁用对应功能。

### 11.3 提示内容

每个缺失提示必须包含：

- 组件名称和用途
- 受影响功能
- 推荐版本和来源
- 下载或离线包大小
- 是否需要联网
- 是否需要管理员权限
- 安装后是否需要重启软件或电脑

### 11.4 安装顺序

1. 先搜索软件旁的 `依赖组件` 目录。
2. 再搜索用户指定的离线安装包。
3. 用户允许联网时才从官方地址下载。
4. 校验签名或哈希后显示确认页。
5. 用户确认后执行安装并记录日志。
6. 安装完成后重新检测并给出结果。

软件不得静默安装、擅自提权、关闭安全软件、修改系统安全策略或自动重启电脑。

## 12. 只读执行与安全设计

- 仅对明确加入项目且用户有权测评的目标发起连接。
- 连接前显示目标地址，批量任务显示目标数量和范围。
- 首次 SSH 连接确认主机指纹；指纹变化立即阻止自动执行。
- 旧算法兼容按设备单独开启，并显示风险和使用记录。
- 凭据使用 DPAPI 加密，内存中只在使用期间保留。
- 命令日志和诊断日志遮盖凭据及敏感参数。
- 软件发出的所有自动命令必须只读。该约束不可由项目设置、插件或普通用户关闭。
- 自动任务只接受正式命令包中的白名单模板和受限参数，不能执行任意文本命令。
- 执行前按对象类型进行结构检查。Shell 拦截写入、删除、重定向、危险管道、命令替换和状态变更工具；PowerShell 拦截状态变更 cmdlet 和脚本；网络设备拦截配置模式及配置命令；数据库使用厂商专用规则拦截 DML、DDL、权限、事务外状态变更、文件写入、动态 SQL、有副作用函数和存储过程；容器拦截创建、启动、停止、重启、删除和配置变更。
- 不能把含有 `SELECT`、`show`、`display` 或 `get` 字样简单认定为安全。正式命令必须经过来源和行为验证。
- 建议使用只读或最小权限账户；即使连接账户具备管理员权限，软件仍执行相同的只读限制。
- 执行前向用户展示本次命令清单，执行后记录实际命令、模板版本、参数、输出和安全检查结果。
- 自动采集界面不提供可绕过白名单的自由命令输入。外部人工终端与自动采集明确分离，并提示其不受本软件只读策略保护。
- 所有外部工具记录版本和哈希，防止被同名文件替换。

## 13. 错误处理

错误分为环境、连接、识别、命令、证据和存储六类。每个错误向用户展示简短说明和处理建议，完整技术信息写入脱敏日志。

- 网络中断：保留已完成命令，可从失败处重新连接并继续。
- 认证失败：停止当前设备，不在无上限循环中重复密码。
- 分页识别失败：停止该命令并保留原始输出，标记人工处理。
- 命令不支持：标记为不兼容，不把错误输出保存为成功证据。
- 识别不确定：停止加载完整命令包，保存识别依据并提示人工选择。
- 安全检查拒绝：不发送命令，显示拒绝原因并记录命令包问题。
- 磁盘空间不足：停止生成新证据，保护已写入文件。
- 截图失败：保留原始输出并允许稍后重新生成截图。
- 用户停止：完成当前安全收尾，记录未执行项目。

## 14. 界面结构

主导航包括：

- 首页：最近项目、待处理问题和组件状态
- 客户与项目：项目资料和证据目录
- 设备：设备列表、连接资料和识别结果
- 采集任务：命令清单、实时状态和安全停止
- 证据：原始输出、截图、备注和完整性校验
- 命令库：正式命令包、待校验命令和版本记录
- 组件中心：环境检测、离线包和安装日志
- 设置：界面、目录、安全和诊断选项

采用左侧导航、顶部项目上下文和主工作区。常用操作显著显示，高级参数折叠。浅色和深色模式均需保证可读性。

## 15. 测试策略

### 15.1 自动测试

- 命令匹配和版本范围
- 命令包导入、校验、锁定和回滚
- 提示符、分页、超时和状态机
- 文件名清理、目录生成和重复执行
- 输出脱敏和 SHA-256 校验
- 依赖状态与安装结果解析
- 自动识别可信度、冲突和多实例处理
- 危险命令拒绝，覆盖网络设备、Shell、PowerShell、SQL、服务和容器状态变更

### 15.2 集成测试

使用本地模拟 SSH/Telnet 服务重现成功、错误密码、主机指纹变化、长输出、分页和网络中断。使用脱敏的真实输出样本验证厂商解析器。

### 15.3 发布测试

至少覆盖 Windows 10 和 Windows 11、管理员与普通用户、联网与离线、中文路径、U 盘运行、只读目录、低磁盘空间和高 DPI。

真实设备兼容性按厂商、型号和版本形成测试矩阵。只有通过矩阵验证的组合才能出现在正式支持清单中。

## 16. 首版验收标准

首版在以下条件全部满足时才算形成可用闭环：

1. 能建立客户和项目并选择证据目录。
2. 能添加单台设备并安全保存凭据。
3. 能通过 SSH 密码和私钥方式连接一个受支持的测试目标。
4. 能自动识别一个网络设备或服务器，并在不确定时要求用户确认。
5. 能在测试主机上发现一个本地或容器化数据库实例，或者明确提示人工补充信息。
6. 能从锁定版本的命令包生成只读任务。
7. 能拒绝测试集中全部已知修改、破坏和有副作用命令，并保证拒绝命令未发送到目标。
8. 能连续执行命令并正确处理至少一种分页方式。
9. 能保存原始输出、执行记录和多页截图。
10. 能在失败、中断和重新执行后保持记录可追溯。
11. 能检测缺失组件并完成一次离线或联网辅助安装流程。
12. 能在 Windows 10 和 Windows 11 上完成相同核心流程。
13. 绿色版和安装版体积均有构建报告，超出目标时可选组件已拆分。

## 17. 分阶段交付

### 第一阶段：最小闭环

完成项目、设备、SSH、基础命令包、任务执行、原始输出、截图、证据目录和启动检查器。

### 第二阶段：连接与命令扩展

加入 Telnet、串口、跳板机、PowerShell/WinRM、命令导入校验和首批网络设备适配。

### 第三阶段：数据库与中间件插件

按实际测评优先级逐个加入数据库和中间件适配包，先验证版本矩阵，再发布支持声明。

### 第四阶段：效率增强

加入批量导入、并发任务控制、证据复核、模板导出和命令包更新管理。并发默认保守，避免对客户网络和设备造成压力。

## 18. 关键产品决定

- 轻量和离线优先于使用体积较大的最新界面运行时。
- 现代视觉通过 WPF 主题和自有样式实现，不牺牲清晰度和可访问性。
- PuTTY/Plink 作为协议能力，不作为产品界面。
- 自动补全命令是待验证建议，不是未经确认即可执行的事实。
- 自动识别优先，人工选择兜底；低可信结果不自动执行完整命令包。
- 软件所有自动命令永久限制为只读查询，不提供关闭该限制的普通设置。
- 用户可后期补充版本命令，但新命令必须先校验再发布。
- 缺失环境允许辅助安装，但每次安装必须由用户确认。
- 支持范围以验证矩阵为准，不使用无法证明的“全品牌、全版本”宣传。
