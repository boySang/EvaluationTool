# SSH 指纹确认与无命令连接测试设计

## 1. 目标

为已保存的 SSH 设备提供第一个真实连接闭环：自动探测主机指纹，由用户人工确认，再使用 DPAPI 保护的密码测试登录。连接测试不执行任何远程命令，不作设备识别，不收集测评证据。

## 2. 产品范围

### 2.1 本期实现

- 设备资料增加必填的 SSH 用户名，按设备保存。
- 选中已保存设备后可启动“测试 SSH 连接”。
- 首次连接先探测 SSH 主机密钥算法和完整指纹，不读取设备密码。
- 界面显示主机、端口、算法、指纹、探测时间和核对提示，由用户明确确认。
- 确认后固定指纹，再使用加密保存的密码执行无命令登录测试。
- 后续测试必须重新校验已固定指纹；指纹变化时立即阻断并保留原指纹。
- 自动探测失败时，允许用户输入从客户管理员或设备控制台取得的完整指纹，但仍需人工确认。

### 2.2 本期不实现

- Telnet、串口、WinRM、跳板机和私钥登录。
- 设备厂商、型号、操作系统、数据库或中间件自动识别。
- 任何查询命令、自由命令、交互式终端或证据截图。
- 指纹变化后的一键覆盖；重新确认将在后续独立流程中实现。

## 3. 交互流程

### 3.1 保存设备

设备页在现有“设备名称、主机地址、SSH 端口、连接密码”基础上增加“SSH 用户名”。用户名只允许 SSH 安全字符，不接受空白、控制字符、命令分隔符或超长内容。

### 3.2 首次连接

1. 用户在“当前项目设备”中选中设备。
2. 点击“测试 SSH 连接”。
3. 软件检查受信任 Plink 组件；组件不可用时禁止继续。
4. 软件仅向已选主机和端口发起指纹探测，不扫描其他端口或主机。
5. 软件显示完整指纹，并提示用户与客户管理员或设备本地控制台核对。
6. 用户点击“指纹已核对，确认并测试登录”后，软件保存确认记录。
7. 软件解密该设备的密码，使用已确认指纹建立 SSH 会话，不下发命令，然后立即关闭。
8. 界面显示连接成功，或显示结构化失败原因。

### 3.3 已确认指纹的后续连接

- 每次登录测试前都先执行一次不使用密码的指纹探测，并与已固定指纹比较。
- 指纹匹配后才能读取密码，登录测试同时向 Plink 传递已固定的完整指纹作为第二道校验。
- 观察指纹一致时记录最新验证时间。
- 观察指纹不一致时记录新观察值，保留原固定值，状态改为“指纹变化，已阻断”，不读取密码。

## 4. 架构与组件

### 4.1 核心域

复用现有 `SshEndpointIdentity`、`HostKeyTrust`、`HostKeyTrustState` 和指纹确认服务。新增连接测试请求、结果和失败类别，不向界面暴露原始进程参数或凭据。

### 4.2 Windows 连接适配器

- `PlinkHostKeyProbe` 使用受信任的固定 Plink 路径探测指纹。
- 探测进程不接收密码、私钥或远程命令。
- `PlinkConnectionTester` 仅在本次无密码探测结果与已固定指纹匹配后创建临时密码租约，以批处理方式建立无命令 SSH 会话。
- 所有进程参数使用结构化令牌构造，不拼接 shell 字符串。
- 诊断日志不记录密码、密码文件路径、原始连接字符串或可能包含敏感内容的完整输出。

### 4.3 应用编排

`SshConnectionTestViewModel` 负责状态、按钮可用性、指纹展示、人工确认和错误提示。它只依赖连接测试服务接口，不直接调用 Plink、SQLite 或 DPAPI。

## 5. 数据持久化

在 SQLite 迁移版本 2 中为设备增加：

- `ssh_username`：SSH 用户名，现有设备迁移后为空，补全前不允许连接。
- `host_key_algorithm`：已固定主机密钥算法。
- `host_key_fingerprint`：已固定完整指纹。
- `host_key_confirmed_at_utc`：确认时间。
- `host_key_confirmation_source`：固定值“用户在设备页人工核对确认”。
- `last_host_key_verified_at_utc`：最近匹配验证时间。
- `blocked_host_key_algorithm`、`blocked_host_key_fingerprint`、`blocked_host_key_observed_at_utc`：指纹变化时的观察记录。

迁移必须兼容已创建的项目和设备，不删除已有数据。指纹更新使用乐观并发检查，防止旧探测结果覆盖新确认。

## 6. 状态与错误

界面状态包括：未就绪、可探测、探测中、等待确认、测试登录中、连接成功、失败、指纹变化已阻断。

错误至少分为：

- 未选择设备或 SSH 用户名缺失。
- Plink 组件缺失、被替换或检查失败。
- DNS 解析、网络不通、端口拒绝、连接超时。
- 无法解析完整指纹或服务端不是 SSH。
- 用户取消确认。
- 指纹变化。
- 用户名或密码错误。
- 服务端策略拒绝、算法不兼容或会话建立失败。

用户提示统一使用“发生了什么、可能原因、怎么处理”格式，技术详情保持脱敏。

## 7. 安全边界

- 指纹确认前不得读取或使用设备密码。
- 登录测试不得下发命令、换行、脚本或交互输入。
- 不依赖 PuTTY 注册表主机密钥缓存，不自动接受未知指纹。
- 不提供忽略指纹、信任任意主机或关闭校验的选项。
- 不默认启用弱算法；算法不兼容时仅提示，不自动降级。
- 只连接用户明确保存的主机和端口，不扫描客户网络。
- 密码临时文件继续使用受限 ACL、短生命周期和可验证删除。

## 8. 验收标准

1. 新建设备必须填写 SSH 用户名，现有设备保留且可补全用户名。
2. 首次探测期间不读取密码，不会对远程终端发送命令。
3. 未点击明确确认按钮时，不得开始密码登录。
4. 确认后可使用正确账号测试登录，且远程端不收到任何命令。
5. 错误密码显示身份验证失败，不显示密码、密码文件路径或原始输出。
6. 指纹变化时在读取密码前阻断，原固定指纹不被覆盖。
7. 组件缺失时项目和设备管理仍可使用，只禁用 SSH 连接功能。
8. Windows 10 和 Windows 11 上都能以普通用户运行，无需管理员权限。
9. 自动化测试覆盖迁移、用户名验证、指纹探测解析、确认门禁、指纹不匹配、身份验证失败、取消、超时和日志脱敏。
