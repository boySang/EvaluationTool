# EvaluationTool 体验版测试清单

本文用于在 Windows 10/11 x64 电脑上验证当前体验版的真实能力。请只使用已经获得明确授权的测试设备，不要直接在尚未确认范围的客户生产设备上首次试用。

## 1. 测试范围

本轮只验收以下闭环：

> 完整解压并启动 → 创建项目 → 添加 SSH Linux 设备 → 探测并人工确认主机指纹 → 无命令登录测试 → 内置只读采集 → 证据中心查看；另可独立验证命令草稿安全导入

当前体验版支持 SSH 用户名加密码，或未加密 PuTTY PPK v2/v3 私钥认证；真实自动采集只对“自动识别”或“服务器”类别的 Linux 设备开放。

## 2. 测试前准备

请准备：

- 一台 Windows 10 x64 或 Windows 11 x64 测试电脑。
- 已启用 .NET Framework 4.8 的 Windows 环境。
- 从 GitHub Actions 下载的 `EvaluationTool-Release-windows-x64` 绿色版和 `EvaluationTool-Installer-windows-x64` 安装版。
- 一个当前 Windows 用户可写的空目录，用于完整解压体验包。
- 一个当前 Windows 用户可写的证据目录，建议路径包含中文，用于验证中文路径兼容性。
- 一台已获授权、可通过 SSH 密码或 PuTTY PPK 私钥登录的 Linux 测试服务器。
- Linux 服务器的 IP 或主机名、SSH 端口、最小权限测试账户，以及密码或未加密 PPK 测试私钥。
- 由服务器管理员通过可信渠道提供的 SSH 主机密钥完整指纹，用于独立核对。

## 3. 下载与完整解压

1. 打开 [boySang/EvaluationTool](https://github.com/boySang/EvaluationTool) 的 **Actions** 页面。
2. 进入 `Windows CI`，打开 `main` 分支最近一次绿色成功运行。
3. 在 **Artifacts** 下载 `EvaluationTool-Release-windows-x64`。构建产物默认保留 14 天。
4. 将压缩包完整解压到本地普通目录。
5. 确认根目录至少包含 `AssessmentTool.App.exe`、若干 DLL、`x64`、`依赖组件`、`SHA256SUMS.txt` 和 `版本与使用说明.txt`。

验收结果：

- [ ] 下载来自成功的 GitHub Actions 运行。
- [ ] 软件已完整解压，不是在压缩包内直接运行。
- [ ] `依赖组件\plink.exe` 存在。
- [ ] 根目录存在 `package-size.json`，其中 `WithinTarget` 为 `true` 且 `TargetMaximumMegabytes` 为 `40`。
- [ ] 没有从其他网站替换或补充未知 DLL、EXE。

### 3.1 安装版验证

1. 下载并完整解压 `EvaluationTool-Installer-windows-x64`。
2. 核对其中 `SHA256SUMS.txt` 与 `EvaluationTool-Setup-windows-x64.exe` 的 SHA-256。
3. 使用普通 Windows 用户运行安装程序，不选择管理员权限。
4. 安装完成后通过开始菜单启动软件，再从 Windows“已安装的应用”卸载。

预期结果：

- [ ] 安装程序默认安装到当前用户目录，不弹出管理员授权请求。
- [ ] 当前安装向导使用英文标准控件，安装后的应用主界面仍为中文。
- [ ] 缺少 .NET Framework 4.8 时停止安装，并询问是否打开微软官方下载页面，不静默下载安装。
- [ ] 安装后主程序和 Plink 均存在，开始菜单快捷方式可正常启动软件。
- [ ] 安装 Artifact 的 `package-size.json` 中 `WithinTarget` 为 `true`，上限为 60 MB。
- [ ] 卸载后程序文件被删除，但 `%LocalAppData%\EvaluationTool` 中的本地项目资料和用户选择的证据目录保持不变。
- [ ] 当前体验安装包可能显示“未知发布者”；只接受成功 Actions 运行生成且 SHA-256 一致的文件。

## 4. 启动与本地环境

1. 使用普通 Windows 用户双击 `AssessmentTool.App.exe`。
2. 等待主窗口显示。
3. 关闭软件后再次启动，确认已有本地数据不会导致启动失败。
4. 打开“组件中心”，单击“重新检测”。
5. 另建一份测试副本，暂时移走测试副本中的 `依赖组件\plink.exe`，重新启动后确认仅 SSH 功能被禁用。
6. 单击“选择离线安装包”，先选择一个任意 EXE，确认校验失败且没有生成或替换 `依赖组件\plink.exe`。
7. 再选择与本体验包配套、来自可信发布包的 `plink.exe`；在二次确认框选择“否”，确认文件仍未安装。
8. 重复选择可信文件并确认安装，随后重新检测。

预期结果：

- [ ] 第一次启动可显示主窗口，没有一闪而退。
- [ ] 第二次启动仍可显示主窗口。
- [ ] 不需要管理员权限。
- [ ] “Plink 连接组件”显示可用并通过可信检查。
- [ ] 如果 Plink 缺失或被修改，SSH 功能保持禁用，但项目等其他功能仍可使用。
- [ ] 任意 EXE 或哈希不匹配的 `plink.exe` 被拒绝，提示不包含所选文件的完整本地路径。
- [ ] 选择可信文件后会显示文件名、大小和 SHA-256；二次确认前及取消后均不替换文件。
- [ ] 确认安装后 Plink 显示可用，SSH 功能恢复，目录中没有遗留 `.plink-install-*` 或 `.plink-backup-*` 文件。
- [ ] 程序目录不可写时安装失败并给出重新解压到可写目录的处理建议，不要求静默提权。
- [ ] 软件没有静默下载、安装组件或重启电脑。

若启动失败，先确认完整解压和 .NET Framework 4.8；不要只复制 EXE 到其他目录运行。

## 5. 创建项目

1. 打开“项目”。
2. 输入测试客户名称和项目名称。
3. 将“证据保存目录”设置为准备好的可写目录，建议使用包含中文和空格的路径。
4. 单击“创建并保存项目”。
5. 在“已保存项目”中选中新项目。

预期结果：

- [ ] 项目创建成功并出现在列表中。
- [ ] 重新启动软件后项目仍然存在。
- [ ] 软件没有在未指定的位置创建客户证据。
- [ ] 项目资料保存在当前 Windows 用户的本地数据库中，没有上传提示或网络上传行为。

## 6. 添加 SSH Linux 设备

1. 打开“设备”，单击“添加设备”。
2. 填写设备名称和主机地址或 IP。
3. 设备类别选择“自动识别（推荐）”或“服务器”。
4. 填写 SSH 用户名和端口，选择密码或 PuTTY PPK 私钥认证并提供对应凭据。
5. 进入“保存与连接测试”步骤，核对信息后单击“保存设备”。

预期结果：

- [ ] 设备出现在当前项目设备列表。
- [ ] 保存设备时没有自动连接目标设备。
- [ ] 保存设备时没有执行任何远程命令。
- [ ] 项目数据库中只保存凭据引用，不出现明文密码、私钥正文或私钥原始路径。
- [ ] 密码或私钥由当前 Windows 用户的 DPAPI 保护，界面和普通日志不显示敏感内容。
- [ ] 加密 PPK、OpenSSH/PEM 私钥、超过 256 KB 或异常编码文件会在保存前被明确拒绝。

## 7. SSH 指纹探测与人工确认

1. 在设备列表中选中刚添加的设备。
2. 单击“探测主机指纹”。
3. 记录界面显示的完整算法和指纹。
4. 与服务器管理员通过可信渠道提供的指纹逐字符核对。
5. 只有完全一致时，单击“确认并测试登录”。

预期结果：

- [ ] 首次探测不读取保存的密码。
- [ ] 首次探测不执行 Linux 查询命令。
- [ ] 界面显示完整指纹，而不是只显示缩略值。
- [ ] 确认后软件重新探测指纹，再开始登录测试。
- [ ] 指纹不一致时，软件在读取密码和自动登录前阻止连接。

安全要求：如果无法从可信渠道获得指纹，停止本项测试，不要为了继续流程而盲目确认。

## 8. 无命令登录测试

在指纹确认后观察“确认并测试登录”的结果。

预期结果：

- [ ] 正确密码时登录测试成功。
- [ ] 登录测试期间不执行 `uname`、`cat`、`hostname` 或其他业务查询命令。
- [ ] 只有登录测试成功后，“开始只读采集”才具备执行条件。

建议补充一次错误凭据测试：使用专门测试设备或重新添加同一测试主机，保存错误密码后完成指纹步骤。

- [ ] 错误密码得到明确失败提示。
- [ ] 错误密码不会触发自动采集。
- [ ] 错误提示不泄露密码。

## 9. 只读采集

1. 选择已通过登录测试的 SSH Linux 设备，或明确授权的华为 VRP 测试设备。
2. 打开“采集任务”。
3. Linux 服务器应自动使用“通用 Linux 服务器”适配器；网络设备必须显示适配器选择框，未选择时不能开始采集。
4. 对华为测试设备明确选择“华为 VRP 网络设备”，对 H3C Comware 7.1/9.x 测试设备明确选择“H3C Comware 7/9 网络设备”，并核对范围说明；不得跨厂商或为 Comware 5 选择该适配器。
5. 单击“开始只读采集”。
6. 观察当前阶段、当前命令编号和状态。
7. 如果出现识别候选项，检查识别依据，确认是目标系统后单击“确认并重新识别”。软件必须重新执行低风险识别，不能直接使用旧候选继续。
8. 等待采集完成。

当前实际命令范围：

| 阶段 | 命令 | 用途 |
| --- | --- | --- |
| 识别 | `uname -a` | 读取内核、主机名和架构信息 |
| 识别 | `cat /etc/os-release` | 读取 Linux 发行版和版本信息 |
| 采集 | `hostname` | 读取主机名 |
| 采集 | `cat /etc/login.defs` | 读取系统默认口令期限参数 |

华为 VRP 网络设备当前实际命令范围：

| 阶段 | 命令 | 用途 |
| --- | --- | --- |
| 识别 | `display version \| no-more` | 读取 VRP、设备系列与软件版本，并关闭本次输出分页 |
| 采集 | `display aaa configuration \| no-more` | 读取 AAA 概要与管理员登录失败锁定参数，不读取口令 |

华为 VRP 流程必须先显示低可信候选并要求人工确认；确认时重新执行版本识别，结果一致后才执行 AAA 查询。真实设备测试需记录型号、VRP 版本、账户权限和结果，不能据此宣称所有华为型号均已验证。

H3C Comware 网络设备当前实际命令范围：

| 阶段 | 命令 | 用途 |
| --- | --- | --- |
| 识别 | `display version \| no-more` | 读取 Comware、设备型号和软件版本，并关闭本次输出分页 |
| 采集 | `display password-control \| no-more` | 读取全局口令老化、长度、复杂度、历史和尝试限制，不读取口令 |

H3C Comware 流程同样必须先人工确认候选，再重新识别并执行口令控制查询。真实设备测试需记录型号、Comware 版本、账户角色和结果。

- [ ] 网络设备未选择采集适配器时，“开始只读采集”保持禁用。
- [ ] 更换到另一台网络设备后，上一台设备的适配器选择不会被沿用。
- [ ] 类别与适配器不匹配时，在创建 SSH 会话前返回 `CollectionAdapterTargetMismatch`，不保存伪造证据。

预期结果：

- [ ] 自动采集没有自由命令输入框。
- [ ] 实际执行的命令只来自上表中的当前内置命令。
- [ ] 识别不可靠时先要求人工确认，确认前不执行完整采集命令。
- [ ] 在待确认候选出现后关闭并重启软件，重新选择同一设备仍能恢复候选；连接组件或主机指纹未就绪时确认按钮不可用。
- [ ] 对恢复的旧候选执行确认时，软件重新运行低风险识别；当前类别、厂商、型号、版本、依据或可信度不一致时停止，不执行完整采集命令。
- [ ] 设备页在采集后显示最近一次设备身份、版本、识别依据、可信度和是否人工确认；关闭并重新启动软件后记录仍然存在。
- [ ] 再次识别同一设备时形成新的修订记录，不覆盖先前识别审计记录。
- [ ] 采集完成后显示已完成命令和证据保存成功提示。
- [ ] 每条命令返回后即可在证据目录看到该命令的独立批次，不需要等待整个任务结束。
- [ ] `cat /etc/login.defs` 的结果只代表系统默认参数，软件没有把它自动判定为所有账户的实际口令策略。

安全停止测试可在再次采集时执行：

- [ ] 单击“安全停止”后不再启动新命令。
- [ ] 已经取得的输出仍保留相应执行记录，不以成功结果掩盖停止状态。
- [ ] 任务执行中强制结束测试软件后重新启动，任务被标记为“异常中断”，软件不会自动连接设备或续跑剩余命令。

## 10. 证据文件与证据中心

1. 打开“证据中心”，单击“刷新索引”。
2. 核对设备名称、命令、执行状态、原始输出相对路径和截图数量。
3. 单击“复核文件 SHA-256”，等待本次文件复核完成。
4. 对任意执行记录分别单击“定位原始输出”和“定位首张截图”，确认资源管理器选中当前记录对应的文件。
5. 单击“打开证据目录”，确认 Windows 资源管理器打开的是当前项目持久化保存的证据根目录。
6. 单击“导出离线报告”，选择一个新的 `.html` 文件名。
7. 单击“导出 JSON 清单”，选择一个新的 `.json` 文件名。
8. 单击“导出已验证证据包”，选择一个新的 `.zip` 文件名。
8. 查找“项目名称 / 设备名称 / 测评项 / 执行批次”目录，并核对每条命令对应的证据文件。

预期结果：

- [ ] 每条已执行命令都有单独批次目录。
- [ ] 存在 `原始输出.txt`。
- [ ] 有输出时存在一张或多张 `证据_*.png`。
- [ ] 存在 `执行记录.json`。
- [ ] 截图内容来自同一份原始输出，并显示项目、设备、测评项、命令、时间和页码信息。
- [ ] 长输出可分页为连续截图，不截断为一张不可读图片。
- [ ] `执行记录.json` 记录命令、起止时间、状态、原始输出 SHA-256 和截图 SHA-256。
- [ ] 普通刷新显示“索引完整（未复核文件）”，没有被误解为已经读取磁盘文件。
- [ ] 显式复核后，未修改的原始输出和截图显示“文件与索引 SHA-256 一致”。
- [ ] 临时修改一个测试证据文件后再次复核，界面显示 SHA-256 不一致；恢复文件前保留测试记录。
- [ ] 删除一个测试证据副本后再次复核，界面显示证据缺失，而不是显示通过。
- [ ] 复核过程只读取证据，不修改文件内容和时间戳。
- [ ] 重复采集生成新的批次目录，不覆盖上一次证据。
- [ ] 中文项目名、中文设备名或中文证据路径可以正常保存。
- [ ] 单条证据定位只接受当前项目索引中的唯一相对路径；缺失、重复、越界或含重解析点的路径被阻止。
- [ ] 导出 JSON 包含项目、设备、执行状态、相对路径、SHA-256、文件复核状态及数据库/中间件发现决议。
- [ ] 导出 JSON 不包含密码、私钥、凭据引用、设备登录用户名、主机地址、绝对证据根目录或原始输出正文。
- [ ] 命令、发现摘要、证据摘录或人工决议原因中出现高置信密码、令牌、私钥头或连接字符串时，JSON 和 ZIP 导出均被阻止。
- [ ] 导出前被修改或删除的证据在 JSON 中如实标为不一致或缺失，不被伪装为通过。
- [ ] 目标 JSON 已存在时软件拒绝覆盖，并提示选择新的文件名。
- [ ] ZIP 包含 `项目证据清单.json`、`证据包说明.txt` 和 `evidence/` 下的实际证据文件。
- [ ] HTML 报告可离线打开和打印，页面无脚本、无外部资源，且不显示主机地址、账号、凭据或原始输出正文。
- [ ] ZIP 内不包含绝对路径、凭据文件、本地数据库或未被当前项目索引引用的文件。
- [ ] 文件在复核后、复制过程中发生变化时打包失败，目标 ZIP 和临时清单不会残留。
- [ ] ZIP 内清单与被打包文件来自同一份不可变导出计划，导出过程中新增的执行或人工决议不会混入半份新数据。
- [ ] 任一证据为缺失、SHA-256 不一致、路径不安全或暂时无法读取时，软件阻止生成 ZIP。

如果出现 `待入库.txt`，表示证据文件已经保存，但 SQLite 索引写入失败；本次测试应记为部分失败并保留该目录，不要删除或伪装为成功。

待入库恢复测试：

- [ ] 在证据中心单击“恢复待入库证据”后，只恢复当前项目及当前项目设备的批次。
- [ ] 恢复前逐个核对执行清单、原始输出和截图 SHA-256；任一文件被修改时保留 `待入库.txt` 并报告仍需人工处理。
- [ ] 恢复成功后刷新索引即可看到对应执行记录，且不会连接设备或执行清单中的命令。
- [ ] 对已经成功入库但标记未删除的批次重复恢复不会生成重复记录。

## 11. Linux 主机数据库实例发现

本节用于当前体验包的数据库实例发现闭环。当前范围仅为 PostgreSQL、MySQL、MariaDB，不应据此认定软件已支持所有数据库、所有安装方式或所有版本。

建议准备以下已获授权的测试场景，可按实际条件选择，不要求一台主机同时具备全部场景：

- 本机进程或 systemd 服务方式运行的 PostgreSQL、MySQL 或 MariaDB。
- Docker 中运行的上述一种数据库。
- Podman 中运行的上述一种数据库。
- 本机服务或容器方式运行的 Nginx、Apache HTTP Server 或 Tomcat。
- 未安装 Docker、未安装 Podman，或两者均未安装的普通 Linux 主机。

测试步骤：

1. 按第 6～9 节完成设备保存、SSH 指纹确认、无命令登录测试和 Linux 身份确认。
2. 开始只读采集，观察数据库与中间件候选发现阶段及其实际执行项。
3. 核对候选中的类别、产品、安装或运行方式、容器名称或服务依据、端口和可信度；界面未提供的数据不得自行推断为已识别。
4. 如果发现一个或多个候选，逐项核对后选择“确认此实例”或“不是本次目标”。
5. 完成人工确认后查看任务状态和证据，确认本阶段没有建立数据库直连，也没有执行 SQL。
6. 关闭并重新启动软件，重新选择同一项目后打开“证据中心”，确认人工确认记录仍可查看，且包含设备、产品、版本、发现依据、可信度、确认时间和确认来源。
7. 在未安装 Docker 或 Podman 的主机上重复测试，确认缺少的容器工具被安全跳过，进程和 systemd 发现仍可继续。

预期结果：

- [ ] 发现范围只涉及当前已授权并已登录的 Linux 主机，没有扫描其他主机或网段。
- [ ] 只读取本机进程名称、systemd 服务状态以及 Docker/Podman 的最少量只读元数据。
- [ ] 原始输出和界面不显示数据库密码、令牌或完整敏感连接字符串，也不采集可能包含密码的完整进程启动参数。
- [ ] 候选产品只按当前规则识别 PostgreSQL、MySQL、MariaDB；其他数据库不会被冒充为已支持对象。
- [ ] 同一批输出可形成 Nginx、Apache HTTP Server、Tomcat 候选，不会为中间件重复执行主机发现命令。
- [ ] 软件重启后仍可恢复未决候选；确认、排除和被新批次取代的历史均不覆盖原记录。
- [ ] 发现结果展示识别依据和可信度，存在多个实例或结果不确定时均保留为候选项。
- [ ] 未经人工确认，软件不会把候选实例作为已确认数据库，也不会继续执行数据库命令。
- [ ] 人工确认只保存发现结论和审计记录，不触发数据库直连，不读取数据库凭据，不执行任何 SQL。
- [ ] 软件重启后，证据中心仍能显示此前的数据库人工确认审计；该记录不被错误标记为命令执行证据或 SHA-256 文件复核结果。
- [ ] Docker 不存在时对应检查可跳过，不导致整个任务失败。
- [ ] Podman 不存在时对应检查可跳过，不导致整个任务失败。
- [ ] 发现命令及其输出、状态和证据可在证据中心追溯。

以下情况应记录为失败或待改进，而不是手工标记为通过：输出泄露敏感启动参数；容器工具缺失导致整个任务失败；未确认就自动选择候选实例；自动尝试数据库登录；执行任何 SQL；把未覆盖的数据库品牌显示为已正式支持。

## 12. 命令草稿安全导入

准备一个不超过 1 MB 的 UTF-8 JSON 文件，其中至少包含根字段 `id`、`name`、`version` 和 `commands` 数组。测试文件可故意把一条命令声明为 `Verified`、`isReadOnly: true`，并使用 `rm`、`configure terminal`、`UPDATE` 或服务启停语句验证阻断提示；请勿在文件中放入真实密码或客户敏感数据。

1. 打开左侧“命令库”。
2. 单击“导入 JSON 草稿”并选择测试文件。
3. 核对文件名、导入时间、SHA-256、命令数量、阻断项和发现列表。
4. 关闭并重新启动软件，确认草稿仍然存在。

预期结果：

- [ ] 所有草稿和草稿命令均显示“待校验 · 禁止执行”。
- [ ] 明显修改、破坏或中高风险语句形成阻断项。
- [ ] 原始 JSON 的 SHA-256 被保存并在重启后保持一致。
- [ ] 页面没有发布或执行草稿的按钮。
- [ ] 大于 1 MB、无效 UTF-8、重复 JSON 字段或格式错误的文件得到明确提示且不入库。
